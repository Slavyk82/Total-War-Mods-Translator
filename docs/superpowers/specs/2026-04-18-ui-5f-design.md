# Plan 5f · Settings polish & primitives — Design

**Date:** 2026-04-18
**Status:** Spec — awaiting implementation plan
**Parent arc:** UI redesign Plans 1 → 5e (all shipped on main)
**Predecessor:** `2026-04-18-ui-settings-design.md` (Plan 5e)

## 1. Purpose

Plan 5f consolidates the seven deferred follow-ups left at the close of Plan 5e. It is **not** a new design direction: no new screens, no new layouts, no new user-visible workflows. The plan closes out tokenisation debt, extends two primitives whose API gaps forced ad-hoc workarounds during retokenisation, and deduplicates code that accumulated during the Plan 5e sprint.

No further UI redesign plans are planned after 5f. Future retokenisation sweeps (if any) would be standalone chores outside the redesign spec tree.

## 2. Scope

### 2.1 In scope (7 tasks)

1. Extend `TokenTextField` with `obscureText`, `autofocus`, `maxLength`, `prefixIcon`; migrate 4 workaround callsites back to the primitive.
2. Add `TwmtThemeTokens.fontDisplayStyle` getter; migrate ~15 repo-wide `fontDisplayItalic ? FontStyle.italic : FontStyle.normal` ternaries.
3. Decompose `llm_models_list.dart` (401 LOC) and `model_management_dialog.dart` (470 LOC) via targeted primitive extraction under `features/settings/widgets/llm/`.
4. Retokenise `FluentToast` (4 types) by adding semantic tokens; unify the two extant `fluent_toast.dart` files into one canonical widget + façade.
5. Extract `_saveApiKey(code, ctrl)` helper in `llm_providers_tab.dart`; replace the 5 duplicated `_save{Provider}ApiKey` methods.
6. Debounce the rate-limit slider `onChanged` in `llm_providers_tab.dart` (300ms), collapsing bursts of ~49 writes per drag.
7. Add a `filled` variant to `SmallTextButton`; migrate the bespoke "Restart Now" filled button in `backup_section.dart`.

### 2.2 Out of scope

- Any new screen, route, or navigation change.
- `FluentToast` API changes beyond colour tokenisation and file merger (callsites already pass `type`, `message`, `duration`; signature unchanged).
- Structural refactor of `llm_models_list.dart` / `model_management_dialog.dart` beyond primitive extraction — no behavioural change, no state-management reshuffle.
- The plan-level follow-ups carried from earlier plans (activity feed staleness, `projectName: null` payload, unsaved inspector edits, dynamic encoding, status dot a11y, `ActionsCellRenderer` dead code, spec/doc reconciliation, `SidebarUpdateChecker` overflow, `_NavItemTile` InkWell, `StatusPill.trailing`). Those remain open as standalone chores.

### 2.3 Non-goals

- No tokenisation audit of widgets outside the 7 listed tasks.
- No new test harness; reuse the existing `golden_toolkit` + `ThemeTestHelpers` setup from Plans 1–5e.
- No schema, service, or repository change. `FluentToast` token additions are the only changes to `TwmtThemeTokens`.

## 4. Primitives

### 4.1 `TokenTextField` (extended)

```dart
const TokenTextField({
  // existing ...
  this.obscureText = false,
  this.autofocus = false,
  this.maxLength,
  this.prefixIcon,
});
```

All four are optional, default-inert. No existing callsite breaks. The four additions are pass-through to the wrapped `TextFormField` / `TextField` with no styling override beyond existing token application.

### 4.2 `TwmtThemeTokens.fontDisplayStyle`

```dart
FontStyle get fontDisplayStyle =>
    fontDisplayItalic ? FontStyle.italic : FontStyle.normal;
```

Trivial derived getter. Migration is mechanical (regex).

### 4.3 Semantic colour tokens (for `FluentToast` and future consumers)

4 new pairs on `TwmtThemeTokens`, distinct per theme:

| Token | Atelier (dark, warm) | Forge (dark, cool) |
|---|---|---|
| `successFg` | ~#7EC97E | ~#6FBF6F |
| `successBg` | ~#1E2A1C | ~#1A2A1A |
| `errorFg` | ~#E88A8A | ~#E37B7B |
| `errorBg` | ~#2E1C1C | ~#2A1818 |
| `warningFg` | ~#E8C870 | ~#E8C06B |
| `warningBg` | ~#2C2518 | ~#2A2216 |
| `infoFg` | ~#7FB8E8 | ~#75B0E0 |
| `infoBg` | ~#1C2530 | ~#18222E |

Exact hex values to be finalised during implementation against the existing `TwmtThemeTokens` palette (the table above is indicative). Both themes already expose comparable semantic colours via `tokens.accent`, `tokens.danger`, etc.; the new tokens follow the same override-per-theme pattern.

### 4.4 `SmallTextButton.filled`

```dart
const SmallTextButton({
  // existing ...
  this.filled = false,
});
```

When `filled: true`: background = `tokens.accent`, foreground = `tokens.accentFg` (or current foreground inverted). Outlined remains the default. No new class created.

## 6. Exit criteria

- `flutter analyze` clean (0 new lint; existing 35 pre-existing lints untouched).
- Test count roughly 1309 + ~13 new widget tests, failure count stays at 14 (pre-existing `SidebarUpdateChecker` overflows per Plan 2 follow-ups).
- `git grep "fontDisplayItalic ?"` returns 0 results after T2.
- `lib/widgets/fluent_toast.dart` deleted; all imports resolve against `lib/widgets/fluent/fluent_toast.dart`.
- No newly-introduced hex literals in any file touched by Plan 5f (all colours via `context.tokens`).
- Rate-limit slider single-drag produces exactly 1 settings write (verified by unit test).

## 7. Risks

- **Token palette extension**: adding 8 tokens to `TwmtThemeTokens` is mechanical but crosses two themes. Risk of forgetting one; mitigated by existing test that asserts theme extensions are symmetric.
- **FluentToast file merger**: 60+ callsites import from either the widget file or the façade. A deferred import cleanup could leave stale imports that compile but reference the deleted file. Mitigation: after deletion, `dart analyze` must report 0 unresolved imports.
- **LLM dialog decomposition**: 900 LOC across 2 files means primitive boundaries may not be obvious. Risk of over-decomposition. Mitigation: aim for 3–5 extractions max; anything beyond that is deferred to a future plan.

## 8. What this plan does not do

- Does not retokenise other shared primitives (e.g., `FluentToast` being shared is the catalyst; there is no analogous sweep of, say, `FluentDialog` in scope).
- Does not introduce new semantic token categories beyond success/error/warning/info.
- Does not unify the outlined vs filled button story across the whole app — only `SmallTextButton` gets a `filled` param.
- Does not audit `TokenTextField` beyond the 4 named props; any further API gap is deferred.
