# Design : préfixe de pack configurable

**Date :** 2026-06-09
**Statut :** Validé (brainstorming)

## Contexte

Les outils qui génèrent des packs Total War utilisent un préfixe codé en dur
`!!!!!!!!!!` (10 points d'exclamation) en tête des noms de fichiers `.pack` et
des `.loc` internes. Ce marqueur force l'ordre de chargement : il garantit que
le pack de traduction se charge en priorité et **écrase** le texte d'origine
du jeu.

Format actuel des noms générés :

- Pack mod : `!!!!!!!!!!_{lang}_twmt_{nom_pack_original}.pack`
- Pack traduction de jeu : `!!!!!!!!!!_{lang}_twmt_game_translation.pack`
- `.loc` interne : `text/db/!!!!!!!!!!_{lang}_twmt_{xxx}.loc`
- Préfixe de compilation (déjà éditable par compilation) :
  `!!!!!!!!!!_{lang}_compilation_twmt_`

**Objectif :** permettre à l'utilisateur de modifier ce préfixe par défaut
depuis Settings/General. La valeur configurée doit se refléter dans tous les
outils de génération de packs, y compris le nommage des `.loc` à l'intérieur
des packs.

## Décisions de design

| Décision | Choix retenu |
|---|---|
| **Portée éditable** | Tout le préfixe **avant** `_{lang}_` (le token `!!!!!!!!!!`). La structure `_{lang}_twmt_…` reste fixe. |
| **Détection vs génération** | **Génération uniquement.** La détection/parsing des fichiers existants continue de reconnaître le standard `!!!!!!!!!!_`. |
| **Validation** | Aucune contrainte forte. On retire seulement les caractères interdits dans les noms de fichiers Windows. Vide autorisé. |
| **Acheminement** | Paramètre explicite : l'orchestrateur lit le préfixe depuis `SettingsService` et le passe en argument aux méthodes de génération. |
| **Warning UI** | Avertissement axé **ordre de chargement** (le vrai risque). |

### Pourquoi « génération uniquement » pour la détection

Beaucoup de mods communautaires utilisent le standard `!!!!!!!!!!_FR_`. Si on
basculait la détection sur le préfixe configuré, l'import de ces mods pourrait
casser. La détection reste donc sur le standard.

### Pourquoi le warning porte sur l'ordre de chargement (et non le scan)

Investigation : `lib/services/mods/pack_file_scanner.dart:107-108` exclut les
packs de traduction TWMT du scan via le marqueur **`_twmt_`**, pas via le
préfixe `!!!!!!!!!!` :

```dart
// Skip packs generated by TWMT (translation packs contain '_twmt_' marker)
if (packFileName.toLowerCase().contains('_twmt_')) {
```

Comme seul le préfixe **avant** la langue est éditable, le marqueur `_twmt_`
(situé après la langue) reste toujours présent dans les noms générés. Le scan
continue donc d'exclure correctement les traductions : le scénario de
mauvaise détection ne se produit pas.

Le risque réel d'une mauvaise valeur de préfixe est l'**ordre de chargement** :
un préfixe affaibli (sans `!`, ou triant « avant » d'autres mods) peut empêcher
la traduction d'écraser le texte d'origine en jeu. Le warning porte sur ce
risque.

## Architecture

### 1. Le réglage (donnée)

- Nouvelle clé `SettingsKeys.packPrefix = 'pack_prefix'`.
- Valeur par défaut : `!!!!!!!!!!` (le marqueur seul, sans `_` final — le code
  ajoute déjà `_${lang}`).
- Constante de défaut centralisée :
  `SettingsKeys.defaultPackPrefixValue = '!!!!!!!!!!'`.
- Helper `SettingsService.getPackPrefix()` → renvoie la valeur stockée ou le
  défaut.
- Stockée dans la table `settings` existante (clé/valeur typée string).
  **Aucune migration de schéma.**

**Sanitisation** appliquée à la saisie UI (avant persistance) : retrait des
caractères interdits dans les noms de fichiers Windows : `/ \ : * ? " < > |`.
Chaîne vide autorisée (le nom généré commencera alors par `_{lang}_…`).

### 2. Acheminement (paramètre explicite)

`ExportOrchestratorService` lit le préfixe via `SettingsService` (injecté ou
via `ServiceLocator`) au début de `exportToPack`, puis le passe :

- → `PackExportUtils.buildPackFileName(lang, src, {String prefix = '!!!!!!!!!!'})`
- → `LocFileServiceImpl.generateLocFilesGroupedBySource(..., {String prefix = '!!!!!!!!!!'})`
  et les helpers internes produisant les chemins `.loc`
  (`text/db/{prefix}_{lang}_twmt_xxx.loc`, ligne metadata `#Loc;1;…`, nom
  `.loc.tsv`, et le `_buildPrefixedLocPath` autour des lignes 492/551/565).

Chaque méthode conserve le **défaut `!!!!!!!!!!`** → aucune régression pour les
appelants non mis à jour ni pour les tests existants.

Les sites concernés (génération) :

- `lib/services/file/pack_export_utils.dart` — `buildPackFileName`
- `lib/services/file/loc_file_service_impl.dart` — lignes 138, 189, 565 (et le
  helper de construction du chemin préfixé)

### 3. Compilations

- `CompilationEditorState.defaultPrefixForLanguage(lang, {String marker = '!!!!!!!!!!'})`
  → `'${marker}_${lang}_compilation_twmt_'`.
- `compilation_editor_notifier` lit le marqueur configuré depuis les settings
  pour **amorcer** le préfixe par défaut d'une nouvelle compilation.
- Le préfixe reste éditable et stocké **par compilation** (champ
  `Compilation.prefix`). On ne change que la valeur d'amorçage. Les
  compilations existantes et le défaut SQL de migration restent inchangés.

### 4. Détection — inchangée

Toutes les fonctions de parsing/détection continuent de reconnaître le
standard `!!!!!!!!!!_` :

- `lib/services/file/models/localization_file.dart` :
  `extractLanguageCode`, `extractBaseName`, et la détection
  `startsWith('!!!!!!!!!!_')`.
- `lib/services/file/localization_parser_impl.dart` (lignes 332-346).
- `lib/services/file/tsv_localization_parser.dart` (regex langue).

**Exception (re-traitement de nos propres fichiers) :** le « strip du préfixe
existant » dans `LocalizationFile.generatePrefixedFileName` reconnaîtra à la
fois le standard `!!!!!!!!!!_` **et** le préfixe configuré, afin de ne pas
double-préfixer un fichier déjà généré avec un préfixe custom. La partie
**génération** de cette méthode utilise le préfixe configuré (paramètre
optionnel, défaut `!!!!!!!!!!`).

### 5. UI — Settings/General

Nouvelle section `lib/features/settings/widgets/general/pack_prefix_section.dart`,
ajoutée à `general_settings_tab.dart` (après une section existante, placement à
confirmer pendant le plan).

Contenu :

- En-tête de section (via `SettingsSectionHeader`).
- **Encart d'avertissement** (icône + texte) : réglage déconseillé ; modifier
  le préfixe peut compromettre l'ordre de chargement et empêcher la traduction
  d'écraser le texte d'origine du jeu. Texte i18n.
- Champ texte (valeur initiale = setting courant).
- Bouton **« Réinitialiser au défaut »** → remet `!!!!!!!!!!`.
- **Aperçu live** sous le champ, recalculé à la saisie :
  - `{prefix}_fr_twmt_mod.pack`
  - `text/db/{prefix}_fr_twmt_text.loc`

Provider : ajout de `packPrefix` à la map `GeneralSettings.build()` et d'une
méthode `updatePackPrefix(String value)` qui sanitise puis persiste via
`SettingsService.setString`, puis `invalidateSelf`.

**i18n :** nouvelles chaînes (titre de section, warning, label du champ, hint,
préfixe d'aperçu, bouton reset) ajoutées au namespace `settings` pour les
**10 langues** (`de, en, es, fr, ko, pt, pt-BR, ru, tr, zh`), puis
régénération slang (`strings_*.g.dart`).

### 6. Tests

- **Unit** :
  - `PackExportUtils.buildPackFileName` avec préfixe custom et avec défaut.
  - Génération `.loc` (`LocFileServiceImpl`) : chemin interne et nom `.loc.tsv`
    avec préfixe custom.
  - Sanitisation : caractères interdits retirés ; vide accepté.
  - `CompilationEditorState.defaultPrefixForLanguage` avec marqueur custom.
- **Provider** : `updatePackPrefix` persiste puis relit la valeur attendue.
- **Détection (non-régression)** : un `.loc` au standard `!!!!!!!!!!_FR_` reste
  détecté (`extractLanguageCode`) même lorsqu'un préfixe custom est configuré.

## Hors périmètre (YAGNI)

- Pas de refactor du nommage en service dédié (`PackNamingService`).
- Pas de bascule de la détection vers le préfixe configuré.
- Pas de migration des compilations existantes.
- Pas de renforcement du scanner via le préfixe configuré (le marqueur
  `_twmt_` suffit).
